## 一、摘要

本專題的目標為實作一個 **Small-C 互動式解譯器**，使我們能從詞法分析、語法分析、符號表管理到程式執行流程，完整理解解譯器的運作原理。

Small-C 保留 C 語言的重要特性，如變數、陣列、指標、控制結構與函式呼叫，並簡化過於複雜的語言機制，使其兼具教學價值與實作可行性。

本專題實作的 Small-C 系統主要分為兩大部分：

1. **互動式操作環境**
   提供使用者指令介面與程式管理功能，包括：

   * `APPEND`
   * `LIST`
   * `EDIT`
   * `DELETE`
   * `INSERT`
   * `SAVE`
   * `LOAD`
   * `CHECK`
   * `RUN`
   * `TRACE`
   * `VARS`
   * `FUNCS`

2. **語言處理核心**
   負責詞法分析、語法分析、符號表管理、記憶體模擬以及程式執行。

使用者可以在終端機中逐步輸入、編輯、檢查並執行 Small-C 程式。

在語言功能方面，本系統支援：

* 整數與字元型別
* 變數宣告
* 陣列
* 指標
* 函式定義與呼叫
* `if / else`
* `while`
* `for`
* `do-while`
* `break`
* `continue`
* `return`
* 內建函式，例如 `printf`、`strlen`、`abs`、`max`、`min`、`sqrt`

系統也支援語法錯誤與執行期錯誤偵測，能在發生錯誤時回報對應訊息，以協助除錯與程式驗證。

在測試與驗證方面，本專題設計多組 Small-C 測試程式與自動化測試案例，驗證變數、陣列、函式、迴圈、錯誤處理與追蹤模式等功能是否能正確執行。

測試結果顯示，本系統已能完成基本 Small-C 程式的解析與執行，並能透過 `TRACE`、`VARS` 與 `FUNCS` 等工具提供執行過程與系統狀態資訊。

透過本專題的開發與測試，我們更深入理解程式語言處理器的核心架構，包括：

* Lexer
* Parser
* Interpreter
* Symbol Table
* Memory Model

也更進一步了解解譯器中各模組之間的互動關係。

## 二、系統架構設計說明

本專題實作一個 Small-C 互動式解譯器，目標是讓使用者可以在終端機中輸入、編輯、檢查並執行簡化版 C 語言程式。

整體系統可以分為兩大部分：

1. **互動式操作環境**
2. **語言處理核心**

### 2.1 互動式操作環境

互動式操作環境主要負責接收使用者輸入的指令，例如：

* `APPEND`
* `LIST`
* `EDIT`
* `DELETE`
* `INSERT`
* `SAVE`
* `LOAD`
* `CHECK`
* `RUN`
* `TRACE`
* `VARS`
* `FUNCS`
* `CLEAR`
* `HELP`
* `ABOUT`
* `QUIT / EXIT`

### 2.2 語言處理核心

語言處理核心則負責將 Small-C 程式碼轉換為可理解的語法結構，並進一步執行程式內容。

執行流程由 `main.py` 作為入口，啟動 `repl.py` 中的互動式介面。使用者在 `sc>` 提示符後輸入指令後，REPL 會判斷輸入內容是環境指令或 Small-C 程式碼。

若是環境指令，例如 `LIST`、`SAVE`、`LOAD`，系統會呼叫 `buffer.py` 中的程式緩衝區管理功能。

若是 `CHECK` 或 `RUN`，則會將目前緩衝區中的程式碼交給解譯器核心處理。

---

```text

repl.py
   ↓
buffer.py
   ↓
interpreter.py
   ├── lexer.py
   ├── parser.py
   ├── symtable.py
   ├── builtin.py
   └── errors.py
```

---

### 2.4 資料流程圖

```text
使用者
  ↓
REPL
  ↓
判斷指令類型

  ├── CHECK → Lexer → Parser → 錯誤訊息 / No errors found
  └── RUN → Lexer → Parser → Interpreter
                                  ↓
                    Symbol Table / Memory / Builtin
                                  ↓
                              執行結果
```

## 三、各模組設計與實作細節

本系統採用模組化設計方式開發 Small-C 解譯器，將不同功能拆分成獨立模組，使各部分能分別開發、測試與維護。

這種設計方式讓系統架構更加清楚，也方便後續除錯與功能擴充。

主要模組包含：

| 模組               | 功能說明        |
| ---------------- | ----------- |
| `main.py`        | 系統啟動入口      |
| `repl.py`        | 互動式操作環境     |
| `buffer.py`      | 程式碼緩衝區管理    |
| `lexer.py`       | 詞法分析        |
| `parser.py`      | 語法分析與部分執行邏輯 |
| `interpreter.py` | 解譯器控制核心     |
| `symtable.py`    | 符號表與記憶體模型   |
| `builtin.py`     | 內建函式        |
| `errors.py`      | 錯誤類別與錯誤訊息   |
| `core_test.py`   | 自動化測試       |

---

### 3.1 main.py

`main.py` 作為系統啟動入口，負責初始化解譯器並進入 `sc>` 互動環境。

---

### 3.2 repl.py

`repl.py` 負責實作 REPL，也就是 **Read–Eval–Print Loop**。

使用者透過 `sc>` 提示符輸入指令後，REPL 會先判斷輸入內容屬於環境指令或 Small-C 程式碼，再決定後續處理方式。

REPL 支援的指令包含：

* `HELP`
* `ABOUT`
* `APPEND`
* `LIST`
* `EDIT`
* `DELETE`
* `INSERT`
* `SAVE`
* `LOAD`
* `CHECK`
* `RUN`
* `TRACE`
* `VARS`
* `FUNCS`
* `CLEAR`
* `NEW`

因此 `repl.py` 除了負責輸入解析，也扮演整體功能協調者的角色。

---

### 3.3 buffer.py

`buffer.py` 負責管理程式緩衝區，主要目的是保存使用者輸入的 Small-C 程式碼。

由於系統提供 `APPEND`、`LIST`、`EDIT`、`DELETE`、`INSERT`、`SAVE` 與 `LOAD` 等指令，因此需要一個能模擬文字編輯器的資料結構。

本系統採用「程式碼行列表」方式保存內容，使每一行都具有固定行號，方便進行新增、刪除與修改等操作。

---

### 3.4 lexer.py

`lexer.py` 負責詞法分析。

當系統接收到 Small-C 程式後，Lexer 會逐字元掃描程式內容，並將原始字串切分為 token，例如：

* 關鍵字
* 變數名稱
* 整數常數
* 字元常數
* 字串
* 運算子
* 標點符號

---

### 3.5 parser.py

完成 token 化後，程式會交由 `parser.py` 進行語法分析。

Parser 會解析 Small-C 語法，例如：

* 變數宣告
* 陣列
* 指標
* 函式定義
* `if / else`
* `while`
* `for`
* `do-while`
* `break`
* `continue`
* `return`
* 運算式

本系統採用 **遞迴下降分析法** 實作 Parser，原因是此方法結構清楚、可讀性高，且適合 Small-C 的語法規模。

若語法合法，Parser 會建立對應結構；若語法不合法，則會回報 Parser Error。

---

### 3.6 interpreter.py

`interpreter.py` 負責程式執行。

當使用者輸入 `CHECK` 或 `RUN` 時，Interpreter 會呼叫 Parser 進行分析，並根據分析結果決定是否執行程式。

此外，以下功能也與 `interpreter.py` 密切相關：

* `TRACE ON / OFF`
* `VARS`
* `FUNCS`

因此 `interpreter.py` 不只是負責程式執行，也扮演 REPL 與語言核心之間的重要橋樑。

---

### 3.7 symtable.py

`symtable.py` 負責符號表與記憶體資訊管理，用於保存：

* 變數名稱
* 型別
* 數值
* 陣列大小
* 函式資訊
* 記憶體位址

由於 Small-C 需要支援變數查詢、函式呼叫與作用域管理，因此 Symbol Table 是解譯器運作的重要基礎。

本系統採用 Python 的 dictionary 作為主要資料結構，使查詢與更新效率較高，也方便支援 `VARS` 與 `FUNCS` 等指令。

---

### 3.8 builtin.py

`builtin.py` 負責集中管理內建函式。

目前支援的內建函式包含：

* `printf`
* `strlen`
* `abs`
* `max`
* `min`
* `sqrt`

將內建函式獨立成模組的好處是 Parser 不需要直接處理函式內容，而是透過函式映射方式交由 Builtin 執行。

若未來需要新增函式，也能較容易擴充。

---

### 3.9 errors.py

`errors.py` 負責統一管理錯誤類別與錯誤訊息，包括：

* Lexer Error
* Parser Error
* Runtime Error

這樣不僅能統一錯誤訊息格式，也能讓系統在發生錯誤時維持穩定，不至於直接崩潰。

## 四、符號表資料結構設計

在 Small-C 解譯器中，符號表是相當重要的核心資料結構。

符號表主要功能是記錄程式中所有識別字的資訊，例如：

* 變數名稱
* 函式名稱
* 型別
* 記憶體位置
* 相關屬性

若沒有符號表，解譯器將無法得知變數是否已宣告、函式是否存在，或某個名稱應對應到哪一段記憶體空間。

因此 Symbol Table 可視為整個解譯器在執行過程中的「資料索引中心」。

---

### 4.1 Symbol Table 管理內容

符號表主要管理兩類資訊：

1. Variables
2. Functions

#### Variables

對於變數，系統會記錄：

* 名稱
* 型別
* 記憶體位址
* 數值
* 陣列大小

例如：

```c
int x = 25;
int data[8];
```

系統不僅會記錄變數名稱，也會為其配置對應的記憶體位置。

#### Functions

對於函式，系統會記錄：

* 函式名稱
* 回傳型別
* 參數資訊
* 定義位置
* 是否屬於 built-in function

例如：

```c
int main() {
    return 0;
}
```

或：

```c
printf("Hello");
```

---

### 4.2 Symbol Table 結構示意圖

```text
Symbol Table
│
├── Variables
│   ├── x
│   ├── y
│   └── data
│
└── Functions
    ├── main
    ├── swap
    └── selection_sort
```

---

### 4.3 與 REPL 指令的關係

符號表的設計也與 REPL 指令密切相關。

當使用者輸入 `VARS` 指令時，系統會從 Symbol Table 中列出目前所有變數資訊。

當使用者輸入 `FUNCS` 指令時，系統則會顯示已定義函式與內建函式內容。

因此 Symbol Table 不只是解譯器內部的資料儲存機制，同時也是提供除錯與資訊查詢功能的重要基礎。

## 五、記憶體模型模擬方式

在 Small-C 解譯器中，除了語法分析與程式執行之外，記憶體模型的設計也是系統能否正確運作的重要部分。

由於 Small-C 支援變數、陣列與指標等 C 語言特性，因此解譯器需要模擬類似真實電腦的記憶體配置方式，才能讓程式中的資料具有儲存與存取能力。

---

### 5.1 變數記憶體配置

系統透過 Symbol Table 與記憶體管理機制共同模擬記憶體環境。

當 Parser 解析到變數宣告時，系統會配置對應的記憶體位址，並將變數名稱與位址記錄在 Symbol Table 中，而實際的數值則儲存在模擬記憶體內。

例如：

```c
int x = 25;
char ch = 'Z';
```

系統不只記錄變數名稱與型別，也會將其配置至特定記憶體位置，使後續程式執行時能夠正確讀寫資料。

---

### 5.2 陣列記憶體配置

陣列採用連續記憶體空間配置方式。

例如：

```c
int data[4];
```

系統會為陣列配置一段連續位址，因此：

```c
data[0]
data[1]
data[2]
data[3]
```

都是透過起始位址加上索引值計算取得。

這種方式與真實 C 語言的陣列概念一致，也讓解譯器能夠有效率地處理陣列存取。

---

### 5.3 指標模擬方式

指標功能同樣建立在記憶體模型之上。

指標本身儲存的是位址，而不是資料值。

例如：

```c
int *p = &x;
```

其中：

* `&x` 代表取得變數 `x` 的記憶體位址
* `p` 負責保存該位址
* `*p` 代表根據 `p` 中儲存的位址，取得實際內容

因此當程式執行 `*p` 時，解譯器會先讀取 `p` 所儲存的位址，再前往該位置取得實際內容。

---

### 5.4 執行期錯誤檢查

本系統也利用記憶體模型進行執行期錯誤檢查。

例如當陣列大小不足卻發生越界存取時，系統會進行索引範圍驗證，並回報錯誤訊息：

```text
Array index out of bounds
```

這能避免非法記憶體存取造成系統錯誤，也讓使用者更容易發現程式問題。

## 七、開發過程遇到的困難與解決方法

在開發 Small-C 解譯器的過程中，我們遇到了幾個主要困難，以下說明問題與對應解決方案。

---

### 7.1 指標陣列索引越界問題

在實作 Selection Sort 程式時，發現函式內部使用指標參數存取陣列元素，例如：

```c
arr[i]
```

系統會回報陣列索引越界錯誤。

問題根源在於，當陣列名稱作為指標參數傳入函式時，函式內部的參數 `arr` 被宣告為指標型別：

```c
int *arr
```

其在符號表中記錄的大小為 1，因此存取 `arr[1]` 時，系統誤判為越界。

#### 解決方式

在 `parse_primary`、`_lval_read` 與 `_lval_write` 等方法中，針對指標型別的變數加入特別判斷。

當符號的 `is_pointer` 屬性為真時，先讀取該符號儲存的基底位址，再加上索引值計算實際記憶體位置，而不套用陣列大小限制檢查。

---

### 7.2 do-while 多行輸入偵測問題

在互動模式中，使用者輸入：

```c
do {
    ...
} while (n <= 20);
```

REPL 無法正確判斷輸入是否完整，導致大括號平衡後就直接執行，而忽略後面的 `while` 條件。

#### 解決方式

在多行輸入偵測邏輯中加入正規表達式判斷。

當程式碼以 `do` 開頭時，除了大括號必須平衡之外，還需要確認：

```c
} while (...);
```

已出現，才視為輸入完成。

---

### 7.3 互動模式重複執行 main 的問題

當使用者在互動模式下輸入單行程式碼，例如：

```c
printf("%d\n", x);
```

若之前已透過 `APPEND` 輸入過含有 `main` 函式的程式，解譯器會在執行單行程式碼時重複呼叫 `main`，導致畫面出現多餘輸出。

#### 解決方式

在 `interpreter.py` 的 `run_program` 方法中加入 `auto_run_main` 參數。

互動模式下單行執行時設為：

```python
auto_run_main = False
```

而 `RUN` 指令執行完整程式時設為：

```python
auto_run_main = True
```

使兩種執行情境能正確區分。

---

### 7.4 TRACE 模式只顯示 token 而非完整行內容

初期實作 `TRACE` 功能時，每行追蹤資訊只顯示當前 token 的值，例如：

```text
while
int
```

而不是完整的程式碼內容，例如：

```c
while (b != 0) {
```

因此可讀性較差。

#### 解決方式

在 Interpreter 初始化時，將原始程式碼依行號切割並儲存於 `source_lines` 列表中。

當 `TRACE` 模式啟用時，`_cur_stmt_preview` 方法會根據當前 token 的行號，直接從 `source_lines` 取出對應行的完整內容並顯示，使追蹤資訊更加完整易讀。

此外，子解譯器 `_TokenListInterpreter` 也需要同步繼承 `source_lines`，才能在函式執行時正確顯示追蹤內容。

### 9.2 技術學習與收穫

透過本專題的開發，我們對程式語言處理器的架構有了更深入的理解。

在語法分析階段，我們實作遞迴下降分析，理解到運算子優先順序如何透過函式呼叫層次來表達，以及如何設計能正確處理左值與右值的指定運算解析邏輯。

在記憶體模型方面，我們學習到如何用 Python 的 dictionary 模擬位址空間，並設計出能支援陣列連續配置與指標間接存取的記憶體管理機制。

這讓我們更直觀地理解 C 語言中指標與陣列的底層運作原理，也讓我們對程式執行時的記憶體結構有了更清楚的認識。

在系統整合方面，本專題採用模組化設計，使語言核心與互動環境能夠獨立開發，並透過明確的 API 介面進行串接。

這樣的設計方式不僅降低了模組間的耦合度，也讓除錯與功能擴充更加方便，是一次實踐軟體工程原則的良好經驗。

每一個 bug 的出現與修復，都加深了我們對解譯器運作機制的理解。

### 9.4 未來改進方向

雖然本系統已能正確執行大多數 Small-C 程式，但仍有若干功能可以在未來進一步改進。

首先，目前的 `TRACE` 模式雖然能顯示完整行內容，但格式仍可進一步美化，例如加入縮排顯示或顏色標示，使追蹤資訊更加易讀。

其次，系統目前尚未支援 `switch / case` 語句，這是作業說明中的加分項目，未來可以嘗試實作。

此外，執行期錯誤處理的完善程度也可以提升，例如加入：

* 空指標檢查
* 更詳細的錯誤位置資訊
* 更完整的錯誤分類

最後，本系統的記憶體模型目前採用固定起始位址配置，若未來要支援動態記憶體配置，例如：

```c
malloc()
free()
```

則需要對記憶體管理機制進行較大幅度的重新設計。

這些改進方向提供了後續研究與開發的可能性，也顯示出程式語言處理器設計的深度與廣度。
